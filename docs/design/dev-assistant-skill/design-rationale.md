# 开发助手 Skill — 设计思路

## 1. 为什么 6 个 Agent，而不是更少或更多？

**不用 1 个大 Agent 的原因：**
- 单个 Agent 的 system prompt 会过于臃肿，影响 LLM 遵循指令的准确性
- 不同场景需要的工具权限不同（如 git-workflow 需要 git 操作权限，code-reviewer 不需要）
- 便于独立调优和扩展

**6 个的划分依据：**
- 按开发阶段划分（初始化 → 开发 → 审查 → 测试 → 发版）
- Bug 修复和文档生成是横跨多阶段的独立能力，单独分出来

**不继续拆分的原因：**
- 当前粒度已足够，再拆会造成功能碎片化
- 如"代码审查"和"代码规范修复"合并在 code-reviewer 中更自然

## 2. 为什么 Commands 和 Agents 分离？

- **Agent** = 角色 + 行为规范（系统级约束）
- **Command** = 快捷入口 + 参数约定（用户级接口）

同一个 Agent 可以被多个 Command 触发，也可以被用户自然语言触发（通过 SKILL.md 描述匹配）。Commands 提供了明确、可自动补全的入口。

## 3. 为什么 prompts 独立文件？

- 模板内容较长（文档模板 7 类 + Bug 流程 5 步），直接嵌入 Agent 的 frontmatter 会让文件难以维护
- `{file:}` 语法引用独立文件，Agent 和 Commands 都能复用
- 可以独立更新模板而不动 Agent 逻辑

## 4. 文档结构设计依据

### 设计文档按模块拆分（design/<模块>/5 文件）

机器人应用通常模块化程度高（电机驱动、传感器、通信等），每个模块独立维护设计文档有助于：
- 新增模块时对照模板快速创建
- 修改模块时只改对应文档
- 5 文件覆盖了"看全景（overview）→ 看数据（data-flow）→ 看接口（interface）→ 看思路（design-rationale）→ 看管线（pipeline）"

### 开发日志按日期（dev-log/YYYY-MM-DD.md）

比单一大文件更易查找、不会膨胀、支持未来按时间范围批量处理。

### 待办分 current/backlog

简单两栏看板，curren 是当前迭代正在做的，backlog 是以后再说。不引入复杂的项目管理工具。

### Bug 按编号（bugs/BUG-XXX.md）

便于 git commit 中引用（如"修复 BUG-003"），按编号增长自然排序。

### 废案按方案名（deprecated/<方案名>.md）

记录为什么放弃某个方案，避免未来重新踩坑。按方案名便于搜索。

## 5. 为什么测试不绑定框架？

- 机器人开发中很多代码依赖硬件或运行时环境，传统单元测试框架不适用
- 用户明确说"有时候直接监听进程有时候写测试脚本"
- test-helper 的责任是**判断**该用什么方式，而不是**强制**某一种

## 6. 为什么 commit 是中文格式？

- 用户明确要求
- 格式"改 <模块> 的 <内容> 解决了 <问题>"直观、可检索
- 个人项目不需要遵循国际化团队的 conventional commits 规范

## 7. 为什么是最小配置的 opencode.jsonc？

- `permission.skill: {"*": "allow"}` 确保所有 skill 自动加载，不弹权限对话框
- Agent 和 Command 通过 Markdown 文件的目录约定自动发现，不需要在 JSON 中重复声明
- model 留给用户自填，不预设（避免绑定特定模型提供商）
