# Git 工作流

你用中文和用户沟通。管理项目的提交、版本发布和 changelog。

## Commit 规范

### 格式

```
改 <模块> 的 <内容> 解决了 <问题>
```

### 示例

```
改电机驱动层的状态机逻辑解决了偶发丢步问题
改传感器模块的数据解析函数解决了浮点精度丢失问题
新增通信层的 ZMQ 封装解决了服务间消息传递问题
重构 PID 控制器的参数配置接口解决了参数调整不便的问题
修复启动脚本的环境变量加载问题解决了开机自启失败问题
```

### 规则

- **首词**：改 / 新增 / 重构 / 修复 / 移除 / 更新
- **模块**：功能模块名（如"电机驱动层"、"通信层"、"传感器模块"）
- **内容**：具体改了哪里
- **问题**：这条改动解决了什么
- 如果一次提交涉及多个文件但解决同一个问题，合并为一条 commit
- 不相关的问题分开提交

## Release 流程

1. 运行 `git log --oneline` 获取未 tag 的 commit 列表
2. 按 commit 类型分组（新增/修复/重构/其他）
3. 生成 changelog（中文，见下方模板）
4. 询问版本号确认
5. 询问是否 push
6. 用户确认后执行：
   ```bash
   git tag -a v<版本号> -m "<版本描述>"
   git push origin v<版本号>
   ```

## Changelog 模板

```markdown
# v<版本号> (<日期>)

## 新增
- <commit 信息>

## 修复
- <commit 信息>

## 重构
- <commit 信息>
```

## 安全规则

- 永远不 force push
- push 前必须用户确认
- 不删除已有 tag
- 不修改远程分支历史
