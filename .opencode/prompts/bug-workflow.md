# Bug 修复流程规范

## 五步流程

```
① 定位 → ② 讨论 → ③ 记录 → ④ 方案 → ⑤ 执行
```

## ① 定位（理解问题）

**目标**：找到问题根因，不是表象。

**动作**：
1. 阅读用户描述，提取关键信息（现象、频率、触发条件）
2. 用 Read 工具读取相关代码文件
3. 搜索日志、错误信息、相关配置
4. 输出可能原因列表（按可能性排序）

**输出格式**：
```
🔍 定位分析

已读取文件：
  - src/motor/driver.py
  - src/motor/pid.py

可能原因（按可能性排序）：
1. [最可能] driver.py L45: xxx 原因...
2. [可能] pid.py L28: xxx 原因...
3. [较低可能] 外部因素: xxx
```

**原则**：
- 不猜测，基于代码和事实
- 不确定的地方标注"需要确认"

---

## ② 讨论（和用户澄清）

**目标**：确保理解正确，补充缺失信息。

**动作**：
- 用 question tool 向用户提问（而非自己假设）
- 问题应该聚焦、可回答
- 每次不超过 3 个问题
- 复述你对问题的理解，确认无误

**示例**：
```
💬 需要确认以下信息：
1. 丢步发生在启动瞬间还是运行中？
2. 丢步后电机会自动恢复吗？
3. 有没有特定的负载条件？
```

**原则**：
- 不要一次性问太多问题
- 不要问用户可以自己看代码回答的问题
- 确认理解后再进入记录阶段

---

## ③ 记录（写入文档）

**目标**：把问题正式记录下来，方便后续追溯。

**动作**：
1. 确定 Bug 编号：检查 `docs/bugs/` 目录，取最大编号 +1
2. 按 `prompts/doc-templates.md` 中的 Bug 模板创建文件
3. 写入：问题描述、复现步骤、定位分析结果
4. 更新 `docs/todo/current.md`（如有相关待办）

**文件位置**：`docs/bugs/BUG-<三位数编号>.md`（如 BUG-001.md）

**内容要求**：
- 客观描述，不掺杂主观判断
- 复现步骤足够详细，别人能照着操作
- 定位分析引用代码行号

---

## ④ 方案（提供选择）

**目标**：提供 2-3 个修复方案，让用户决策。

**动作**：
1. 方案 A：最小改动（通常推荐）
2. 方案 B：更彻底但改动更大
3. 方案 C（可选）：其他思路
4. 分析每个方案的优缺点、改动范围、风险

**输出格式**：
```
💡 修复方案

方案 A（推荐）：<标题>
  改动范围：<文件列表>
  优点：<至少 2 条>
  缺点：<至少 1 条>
  风险：低/中/高

方案 B：<标题>
  改动范围：<文件列表>
  优点：<至少 2 条>
  缺点：<至少 1 条>
  风险：低/中/高

建议选 A，因为改动小、风险低。
```

**原则**：
- 不要给用户唯一选项
- 每个方案都该是可行的
- 标明推荐方案和理由

---

## ⑤ 执行（等用户指令后修改）

**目标**：收到用户明确指令后，进行代码修改。

**动作**：
1. 等待用户明确选择方案（如"用方案A"）
2. 用 Read 再次读取目标文件（确认最新内容）
3. 用 Edit 工具精确修改
4. 输出修改摘要
5. 更新 Bug 文档中的修复记录

**禁止**：
- 未经用户确认直接修改代码
- 修改方案之外的文件

**输出格式**：
```
✅ 已修改 src/motor/driver.py:
  L43-L52: 新增相位对准逻辑
  L67: 调整定时器配置

测试建议：重复启停 50 次，确认无丢步。

Bug 文档已更新：docs/bugs/BUG-001.md
```

---

## 额外规则

1. **用户说"直接修"**：跳过 ②③④，读取 → 修改 → 输出摘要
2. **用户说"先记下来"**：只执行到 ③，不进入方案阶段
3. **无法定位**：诚实告知，建议用户提供更多日志或调试信息
4. **不是 Bug 是用法的**：友好解释，给出正确用法示例
